# Group Panel Ideation

This document explores different approaches for the multi-task group panel.

**Scope:** What users SEE when they click on a group with 2-1000 tasks.

**Constraints:**
- Must scale from 2 tasks to 1000 tasks
- Must support various queries: status, IP, name, time, node, etc.
- Must remain consistent and predictable

---

## User Journeys

### Journey 1: "Did my training job work?"

**Context:** User submitted a distributed training job with 8 GPU tasks. Came back after lunch.

**What they need:**
- Quick yes/no: All completed?
- If not: What's still running? What failed?
- Duration: How long did it take?

**Key info:** Status summary, completion progress, total time

---

### Journey 2: "Why did 3 of my 50 preprocessing tasks fail?"

**Context:** User ran parallel preprocessing. Most succeeded, some failed.

**What they need:**
- Which tasks failed?
- Is it the same failure or different?
- What nodes were they on? (Hardware issue?)

**Key info:** Failure details, grouping by failure type, node distribution

---

### Journey 3: "Find the task that processed shard 247"

**Context:** User has 500 parallel tasks. Needs to find one specific task.

**What they need:**
- Search by name
- Quick navigation to that task's details/logs

**Key info:** Searchable task list

---

### Journey 4: "Which tasks are stuck in scheduling?"

**Context:** User's job is taking forever. Some tasks haven't started.

**What they need:**
- Filter by status
- See how long tasks have been waiting
- Identify bottleneck (queue position, resource contention)

**Key info:** Status filter, wait time, scheduling info

---

### Journey 5: "Compare resource usage across my training replicas"

**Context:** User has 8 training replicas. One seems slower.

**What they need:**
- See all tasks with their durations
- Quick comparison view
- Access to metrics/dashboard for each

**Key info:** Duration comparison, node placement, metrics access

---

### Journey 6: "Find tasks running on gpu-node-12"

**Context:** Hardware issue suspected on a specific node.

**What they need:**
- Filter by node
- See status of those tasks
- Check if failures correlate with node

**Key info:** Node filter, status distribution per node

---

### Journey 7: "See tasks that started in the last 5 minutes"

**Context:** Monitoring a running job, want to see recent activity.

**What they need:**
- Sort by start time
- See what's new
- Live updates

**Key info:** Time-based sorting/filtering

---

## Open Questions

1. **Should failures auto-filter by default?**
   - Pro: Most common use case is "what failed?"
   - Con: Might confuse users ("where are my tasks?")

2. **How prominent should the progress bar be?**
   - Large and colorful vs subtle and compact?

3. **Column selection: fixed or customizable?**
   - Fixed: Status, Name, Duration, Node
   - Optional: IP, Start Time, End Time, Exit Code, Resources

4. **Live updates while running?**
   - Auto-refresh vs manual refresh?
   - How to indicate new activity?

5. **Empty states:**
   - No tasks yet (pending)?
   - No matching results (filter too restrictive)?
